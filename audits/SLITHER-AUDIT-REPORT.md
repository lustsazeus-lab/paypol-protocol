## High Severity Findings

### H-01: Unchecked ERC-20 Transfer Return Values
**Detector:** `unchecked-transfer`
**Severity:** High
**Contract:** `SecurityDepositVault.sol`
**Status:** Acknowledged — To be fixed before mainnet

**Description:**
Three functions in `SecurityDepositVault` do not check the return value of ERC-20 `transfer` and `transferFrom` calls:

1. `deposit()` — Line 96: `IERC20(token).transferFrom(msg.sender, address(this), _amount)`
2. `withdraw()` — Line 127: `IERC20(token).transfer(msg.sender, _amount)`
3. `insurancePayout()` — Line 174: `IERC20(token).transfer(_claimant, _amount)`

**Impact:** If the token returns `false` instead of reverting on failure, the contract state would update without actual token movement. On Tempo Moderato Testnet with AlphaUSD (which reverts on failure), this is not exploitable, but it would be dangerous with non-standard ERC-20 tokens.

**Recommendation:**
```solidity
import {SafeERC20} from "@openzeppelin/contracts/token/ERC20/utils/SafeERC20.sol";
using SafeERC20 for IERC20;
// Replace: IERC20(token).transfer(to, amount)
// With:    IERC20(token).safeTransfer(to, amount)
```

**Risk on Testnet:** Low (AlphaUSD reverts on failure)
**Risk on Mainnet:** High (must fix before mainnet deployment)

### H-02: False Positives — PlonkVerifier Assembly Returns (28 findings)
**Detector:** `incorrect-return`
**Severity:** Reported as High — **Actually False Positive**
**Contracts:** `PlonkVerifier.sol`, `PlonkVerifierV2.sol`

**Description:**
Slither flags 28 instances of assembly `return(0, 0x20)` in the PLONK ZK-SNARK verifier contracts. These are auto-generated by the snarkjs/circom toolchain and use low-level assembly for cryptographic pairing operations.

**Why these are false positives:**
- The `return` statements in assembly are the **intended behavior** for PLONK verification
- These contracts are generated by trusted tooling (snarkjs PLONK export)
- The assembly `return` halts execution with the verification result (true/false)
- This pattern is used by all PLONK verifiers across the industry

**Action:** No fix needed. These are correctly implemented ZK verifier contracts.

### M-02: Potential Reentrancy in PayPolStreamV1
**Detector:** `reentrancy-no-eth`
**Severity:** Medium
**Contract:** `PayPolStreamV1.sol`

**Description:**
The `approveMilestone()` function performs an external ERC-20 transfer before updating all state:
```solidity
bool success = IERC20Stream(s.token).transfer(s.worker, milestoneAmount);
// State updates after transfer...
```

**Impact:** On Tempo with AlphaUSD, reentrancy via ERC-20 callback is not possible (AlphaUSD is a standard ERC-20 without callbacks). However, if integrated with ERC-777 or other callback tokens, this could be exploited.

**Recommendation:** Add OpenZeppelin `ReentrancyGuard` or follow checks-effects-interactions pattern:
```solidity
// Update state first
s.milestones[milestoneIndex].approved = true;
s.releasedAmount += milestoneAmount;
// Then transfer
IERC20Stream(s.token).transfer(s.worker, milestoneAmount);
```

**Risk on Testnet:** None (AlphaUSD has no callbacks)
**Risk on Mainnet:** Medium (must fix if supporting arbitrary tokens)

## Recommendations for Mainnet

### Must Fix (Before Mainnet)
1. **SecurityDepositVault**: Use `SafeERC20` for all token transfers
2. **PayPolStreamV1**: Add `ReentrancyGuard` or reorder state updates before external calls
3. **All contracts**: Add zero-address checks in constructors

### Should Fix (Best Practice)
4. Emit events before external calls (CEI pattern)
5. Mark constant/immutable state variables for gas savings
6. Cache array lengths in loops

7. PlonkVerifier assembly returns (false positives)
8. Timestamp usage (intentional for deadline logic)
